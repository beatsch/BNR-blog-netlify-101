Overview

The assessment of our page according to the WCAG 2.0 Guidelines

Accessibility Highlights

Our Web page is accessible in terms of good contrast between the background and the text. This makes it possible for the majority of the visitors to visit our Web page without having any problems. Also, on our Web page purpose of the links can be determined by its name. This prevents visitors from clicking on a link and going to a different page than wanted.

Improvement Plan

1. Perceivable

1.1. Text Alternatives
1.1.1. Non-text content
We should provide a text alternative for the header image. Furthermore, we should have used the <alt> attribute for the <input> which enhances the accessibility of our Web page and it helps to improve search engine optimization (SEO).

1.3 Adaptable
1.3.1. Our Web Page would would have complied to guideline 1.3.1 if we would have had provided an alternative text for the header image since this is the only element that gets lost when using a different stylesheet or even better: not using header image. Instead, HTML and CSS can be used to make a simple logo which does still show up when another or no stylesheet is used. Also, making sure that structure of the page does not get lost (i.e. elements of the page not in the right place anymore) will make our page comply to guideline 1.3.1.
1.3.2. We can include in the "Number 1", "Number 2" and so on for each element in the ordered list and also leave out the <th> elements in the table in the footer, since they are not of real importance. The latter situation applies also on the table with the phones, because you cannot place a <td> element in a different column with a different heading. For instance, the screen size of the phone cannot be placed in a column which should contain the model of the phone. However, we cannot leave out those headings since they are important. Also, the order of the elements of the form is depends on the headers of the table and since we do not leave out those, we cannot change the order of the elements. However, if we would leave out the headers, then we could add labels to each field to make the elements independent from their order. Actually, we use have placeholders in the form fields, but the independency of the order of the elements cannot depend on those placeholders in our opinion.
1.3.3. We can solve the problem discussed in Appendix A by looking for phone images with the same sizes which does not have to be adjusted using CSS. Also, we have to make sure that the header image does not depend on its size or replace it by a simple text logo.

1.4. Distinguishable
There are no further improvements needed to make our Web page more distinguishable as explained in Appendix A.

2. Operable

2.1. Keyboard Accessible
2.1.1. We should have made links and buttons navigable by keyboard by making use code with keyboard shortcuts and linking to this code to our HTML documents.

2.4. Navigable
2.4.1. We do not have blocks of content repeated on multiple pages. However, if we would have had this, then we would have to find a way to make it possible to bypass. We could do this by using script which makes it possible to collapse and expand elements such as the header and the footer. Still, this criterion is not one we can use to base our judgement on of how navigable our site is.

2.4.2. The Web page has a title, so 2.4.2 complies to the Web page.
2.4.3. This criterion does not apply to our Web page, because our Web page cannot be navigated sequentially.
2.4.4. Although the links on our page do not link to existing pages, the purpose of each link on our Web page can be determined by the text only. Thus, our Web page complies to 2.4.4.
2.4.5. This does not apply to us, because we only have one Web page.
2.4.6. Each section of our Web page has a title or heading.
